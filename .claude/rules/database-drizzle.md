# Database & Drizzle

## Transactions (mandatory)

### Multi-write = transaction

Any operation that runs 2+ sequential write queries (insert, update, delete) must use `db.transaction()`.
This applies even when writes target the **same** table. If one query fails, partial state is rolled back.

```ts
// FORBIDDEN — sequential writes without transaction
await db.delete(categories).where(...);
await db.insert(categories).values(newRows);
await db.update(declarations).set({ step: 2 }).where(...);

// CORRECT — wrapped in a transaction
await db.transaction(async (tx) => {
  await tx.delete(categories).where(...);
  await tx.insert(categories).values(newRows);
  await tx.update(declarations).set({ step: 2 }).where(...);
});
```

### Read-then-write = transaction

Any operation that reads data and then conditionally writes based on the result must also use a transaction.
Without it, a concurrent request can read stale data and cause duplicates or conflicts.

```ts
// FORBIDDEN — read-then-write without transaction (race condition)
const existing = await db.select().from(items).where(...);
if (existing.length === 0) {
  await db.insert(items).values(newItem);
}

// CORRECT — atomic read-then-write
return db.transaction(async (tx) => {
  const existing = await tx.select().from(items).where(...);
  if (existing.length === 0) {
    return tx.insert(items).values(newItem).returning();
  }
  return existing;
});
```

**Rule of thumb:** if a procedure has more than one `await db.`/`await tx.` call where at least one is a write, wrap everything in `db.transaction()`.

## No Date computation at module scope

Never compute dates (e.g. `new Date()`, `Date.now()`) at the top level of a module.
Module-scope code runs once at import time and the value becomes stale.
Compute dates inside functions or request handlers.

```ts
// FORBIDDEN
const currentYear = new Date().getFullYear();

// CORRECT
function getCurrentYear() {
  return new Date().getFullYear();
}
```

## camelCase schema properties

Schema properties are always camelCase. The `casing: "snake_case"` config handles DB column mapping.
Never specify explicit column name strings.

## Drizzle Kit workflow (mandatory)

Migrations are generated by Drizzle Kit — **never hand-write or edit migration files**.

### After a schema change

```bash
pnpm db:generate   # generates SQL migration + snapshot in drizzle/
pnpm db:migrate    # applies the migration to the database
```

### Rules

- **Never manually edit** files in `drizzle/` (SQL migrations, `meta/_journal.json`, snapshots).
  If a migration is wrong, delete it and re-run `pnpm db:generate`.
- **Never fabricate UUIDs** in snapshot files — they are generated by Drizzle Kit.
- Use `pnpm db:push` only for local development (syncs schema without creating a migration file).
- In Docker Compose, the `migrate` service runs `pnpm db:push` after the database is healthy — this is for local dev only.
